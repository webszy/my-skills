# App Analytics Skill

## 1. Skill 目标

本 Skill 用于把一次 App 数据分析拆分为可重复、可校验、可追溯的五个阶段：

```text
代码扫描
  ↓
设计数据输入
  ↓
校验输入数据
  ↓
执行 App 分析
  ↓
生成最终报告
```

本文件只负责：

- 判断当前处于哪个阶段
- 确定需要读取的输入
- 调用对应的 workflow
- 检查阶段完成条件
- 管理阶段之间的依赖关系
- 规定全流程的事实边界与输出规则

每个阶段的详细执行规则位于：

```text
workflows/
├── 01-code-scanner.md
├── 02-design-data-inputs.md
├── 03-validate-input-data.md
├── 04-analyze-app.md
└── 05-generate-report.md
```

不要把各阶段的详细提示词重复写入本文件。

---

## 2. 适用场景

当用户提出以下任务时使用本 Skill：

- 扫描 App 项目并梳理页面、跳转、广告、订阅和埋点
- 判断为了分析某个 App 还需要导出哪些平台报表
- 生成需要用户补充的业务信息模板
- 校验 Firebase、AdMob、App Store Connect、Adjust 等数据文件
- 分析用户规模、页面路径、功能使用率、广告表现或订阅转化
- 对比不同版本、日期、国家、渠道或广告策略
- 输出完整的 App 数据分析报告
- 基于已完成的分析继续执行专题分析

本 Skill 不负责：

- 修改 App 业务代码
- 自动新增埋点
- 修改广告或订阅配置
- 直接操作线上数据平台
- 把未经确认的推断写成事实
- 在输入数据不足时伪造指标或结论

默认使用以下项目结构：

```text
<APP_PROJECT_ROOT>/

│
├── workflows/
│   ├── 01-code-scanner.md
│   ├── 02-design-data-inputs.md
│   ├── 03-validate-input-data.md
│   ├── 04-analyze-app.md
│   └── 05-generate-report.md

├── APP_DATA_REPORT_REQUIREMENTS.md
├── APP_USER_INPUT_REPORT.md
├── APP_DATA_VALIDATION_REPORT.md
│
├── data/
│   ├── firebase/
│   ├── admob/
│   ├── appstore/
```

## 4. 核心文件职责

| 文件 | 职责 | 主要来源 | 是否允许人工填写 |
| --- | --- | --- | --- |
| `APP_PROJECT_SCAN_REPORT.md` | 保存代码中可确认的项目事实 | 项目源码 | 否 |
| `APP_DATA_REPORT_REQUIREMENTS.md` | 定义分析所需的平台报表、字段和粒度 | 扫描报告 | 原则上否 |
| `APP_USER_INPUT_REPORT.md` | 保存代码和平台报表无法确认的业务事实 | 用户 | 是 |
| `APP_DATA_VALIDATION_REPORT.md` | 记录已提交数据的完整性、口径和可计算性 | 用户报表与输入表 | 否 |
| `analysis/APP_ANALYSIS_RESULTS.md` | 保存分析计算、证据、异常和初步判断 | 已校验数据 | 否 |
| `analysis/APP_ANALYSIS_REPORT.md` | 面向决策的最终报告 | 分析结果 | 否 |

不同文件必须保持职责独立。

不得把用户人工填写的数据写回代码扫描报告，也不得把分析结论写入数据校验报告。

---

## 5. 全流程状态判断

执行任务前，先检查项目中已有文件和用户当前请求。

### 5.1 阶段判断规则

| 当前状态 | 应执行的 workflow |
| --- | --- |
| 尚无代码扫描报告 | `01-code-scanner.md` |
| 已有扫描报告，但没有数据需求或用户输入表 | `02-design-data-inputs.md` |
| 用户已经提交数据，但尚未完成数据校验 | `03-validate-input-data.md` |
| 数据已经校验并具备分析条件 | `04-analyze-app.md` |
| 已经完成分析结果，需要形成正式交付物 | `05-generate-report.md` |

### 5.2 最小执行原则

默认只执行满足用户当前请求所需的最小阶段。

例如：

- 用户只要求扫描代码，不要自动进入数据设计阶段。
- 用户只要求判断缺少哪些报表，不要直接开始分析。
- 用户要求正式分析时，如果输入尚未校验，应先执行数据校验。
- 用户要求生成最终报告时，如果尚无分析结果，应先执行分析阶段。

只有用户明确要求“执行完整流程”“从扫描到最终报告全部完成”时，才连续执行多个阶段。

### 5.3 阶段回退

出现以下情况时，应回到前置阶段：

| 情况 | 回退阶段 |
| --- | --- |
| 项目代码发生明显变化 | 阶段 1 |
| 新增页面、广告位、产品或埋点 | 阶段 1 或阶段 2 |
| 用户更改分析目标或时间范围 | 阶段 2 |
| 用户重新上传或替换数据文件 | 阶段 3 |
| 数据校验发现口径冲突 | 阶段 2 或阶段 3 |
| 正式报告引用的数据发生变化 | 阶段 4 |

# 6. Workflow 01：代码扫描

执行文件：

```text
workflows/01-code-scanner.md
```

## 6.1 目标

从项目代码中客观识别 App 的产品结构和可观测结构。

## 6.2 输入

- 当前 App 项目源码
- 项目配置文件
- 依赖文件
- 路由、页面、广告、订阅和 Analytics 代码

## 6.3 输出

```text
APP_PROJECT_SCAN_REPORT.md
```

## 6.4 主要内容

- 项目基本信息
- 主要目录结构
- 页面清单
- 页面跳转关系
- 页面关系图
- 广告 SDK 与广告位
- 订阅、内购和 Paywall
- 付费触发点
- Analytics SDK 与事件
- 关键配置
- 外部依赖
- 未确认信息

## 6.5 阶段边界

本阶段只能记录代码中能够确认的事实。

禁止：

- 经营分析
- 数据表现分析
- 产品优劣评价
- 广告策略评价
- Paywall 转化判断
- 虚构页面、事件或触发点
- 推荐新增埋点

## 6.6 完成条件

- 已生成扫描报告
- 核心结论附有文件路径或代码依据
- 不确定内容已标记“未确认”或“不明确”
- 未修改项目代码和配置
- 报告中不包含经营分析结论

# 7. Workflow 02：设计数据输入

执行文件：

```text
workflows/02-design-data-inputs.md
```

## 7.1 目标

根据代码扫描结果，确定后续分析需要哪些平台报表，以及哪些业务信息必须由用户填写或确认。

## 7.2 输入

必需：

```text
APP_PROJECT_SCAN_REPORT.md
```

可选：

- 用户当前分析目标
- 用户已提供的数据文件
- 用户明确说明的平台和数据来源

## 7.3 输出

```text
APP_DATA_REPORT_REQUIREMENTS.md
APP_USER_INPUT_REPORT.md
```

## 7.4 `APP_DATA_REPORT_REQUIREMENTS.md` 应包含

- 已识别的数据来源
- 报表需求总表
- P0 / P1 / P2 优先级
- 每份报表的必需字段
- 时间范围
- 数据粒度
- 分析维度
- 表间关联键
- 可计算指标
- 第一批最小数据集
- 当前数据缺口

## 7.5 `APP_USER_INPUT_REPORT.md` 应包含

- App 与分析范围
- 数据源登记
- 版本和发布时间
- 产品与策略改动
- 页面名称映射
- 事件业务含义
- 功能成功标准
- 广告位映射
- 广告展示策略
- 订阅产品映射
- Paywall 策略
- 投放信息
- 业务目标
- 异常日期
- 数据文件登记
- 用户待办清单

## 7.6 阶段边界

本阶段只设计输入要求，不分析表现。

禁止：

- 根据代码推断实际用户行为
- 根据广告位存在推断广告收入
- 根据 Paywall 存在推断订阅转化
- 把页面进入视为功能完成
- 虚构平台一定存在的字段
- 直接输出优化建议

## 7.7 完成条件

- 所有报表均有唯一报表 ID
- P0 数据已经明确
- 每份报表写明字段、粒度和来源
- 用户输入表已尽可能自动预填
- 未确认内容保留 `【待填写】` 或 `【待确认】`
- 已生成用户最终待办清单

# 8. Workflow 03：校验输入数据

执行文件：

```text
workflows/03-validate-input-data.md
```

## 8.1 目标

检查用户提交的报表和业务输入是否足以支持后续分析，并明确每个指标当前是否可计算。

## 8.2 输入

必需：

```text
APP_PROJECT_SCAN_REPORT.md
APP_DATA_REPORT_REQUIREMENTS.md
APP_USER_INPUT_REPORT.md
```

数据文件：

```text
data/*
```

## 8.3 输出

```text
APP_DATA_VALIDATION_REPORT.md
```

## 8.4 校验范围

### 文件级校验

- 文件是否存在
- 文件是否可读取
- 文件格式与字符编码
- 日期范围
- 数据行数
- 重复文件
- 空文件或截断文件

### 字段级校验

- 必需字段是否存在
- 字段类型是否正确
- 日期格式
- 数值格式
- 币种
- 时区
- 空值比例
- 重复行
- 异常值

### 映射校验

- App ID
- App 版本
- 页面名称
- 事件名称
- 广告位 ID
- 广告位名称
- 产品 ID
- 渠道和 Campaign

### 口径校验

- 活跃用户定义
- 新增用户定义
- 页面曝光定义
- 广告请求、匹配、展示和收入定义
- Paywall 展示和购买定义
- 订阅收入和退款定义
- 时间范围和时区
- 去重单位

## 8.5 可计算性分类

每个指标必须标记为：

- 可直接计算
- 可在补充映射后计算
- 可在补充报表后计算
- 只能部分计算
- 当前无法计算
- 不适用

## 8.6 阶段边界

数据校验阶段不输出经营结论。

允许说明：

- 哪些字段缺失
- 哪些数据冲突
- 哪些日期不完整
- 哪些指标无法计算
- 哪些映射需要确认

禁止说明：

- 指标表现好坏
- 问题原因
- 产品优化方向
- 广告或订阅策略建议

## 8.7 完成条件

- P0 报表均有明确状态
- 关键字段和映射已经检查
- 每个核心指标均有可计算性判断
- 阻塞项已单独列出
- 明确第一轮分析是否可以启动

# 9. Workflow 04：执行 App 分析

执行文件：

```text
workflows/04-analyze-app.md
```

## 9.1 目标

在数据校验通过的前提下，对 App 的用户、页面、功能、广告、订阅、投放和版本变化进行计算与诊断。

## 9.2 输入

必需：

```text
APP_PROJECT_SCAN_REPORT.md
APP_DATA_REPORT_REQUIREMENTS.md
APP_USER_INPUT_REPORT.md
APP_DATA_VALIDATION_REPORT.md
data/*
```

## 9.3 输出

```text
analysis/APP_ANALYSIS_RESULTS.md
```

专题分析可输出到：

```text
analysis/topics/<TOPIC_NAME>.md
```

## 9.4 分析模块

根据项目实际数据选择适用模块：

1. App 整体规模与趋势
2. 新增、活跃与留存
3. 启动、会话和使用时长
4. 页面曝光与页面渗透率
5. 页面路径与用户流失
6. 功能使用率与完成率
7. 功能漏斗
8. 广告请求、匹配、展示和收入
9. 人均广告展示次数
10. 广告位、国家和版本表现
11. Paywall 曝光与购买路径
12. 试用、首购、续订、退款和收入
13. 投放成本与 ROAS
14. 版本和策略变化对照
15. 异常日期和数据质量影响

不适用或无法计算的模块必须明确跳过，不能补造数据。

## 9.5 每项分析的证据结构

每个分析结论至少包含：

```text
分析问题
→ 使用的数据文件
→ 时间范围
→ 过滤条件
→ 指标口径
→ 计算结果
→ 对比基准
→ 事实发现
→ 可能解释
→ 置信度
→ 尚缺证据
```

必须区分：

- 数据事实
- 基于数据的推断
- 尚未验证的假设

## 9.6 分析优先级

优先处理：

1. 直接影响收入或成本的异常
2. 主流程中的大规模流失
3. 核心功能使用或完成问题
4. 广告展示和变现问题
5. 订阅和 Paywall 转化问题
6. 留存和回访问题
7. 次要页面和低流量功能

优先级必须基于影响范围和数据证据，不得只依据主观感受。

## 9.7 阶段边界

- 不允许分析校验报告明确标记为不可用的数据。
- 不得把相关性表述为确定因果。
- 不得隐藏样本量过小、日期缺失或口径冲突。
- 不得为了形成结论而补造数据。
- 对无法验证的原因，只能写成假设。

## 9.8 完成条件

- 所有核心结论可追溯到数据文件和口径
- 事实、推断和假设已明确区分
- 关键异常有影响范围
- 重要结论有对比基准
- 已记录数据限制
- 已形成可供报告阶段使用的分析结果

# 10. Workflow 05：生成最终报告

执行文件：

```text
workflows/05-generate-report.md
```

## 10.1 目标

把分析阶段的计算和发现整理成面向产品、运营或管理决策的正式报告。

## 10.2 输入

必需：

```text
analysis/APP_ANALYSIS_RESULTS.md
APP_DATA_VALIDATION_REPORT.md
APP_USER_INPUT_REPORT.md
```

必要时回查：

```text
APP_PROJECT_SCAN_REPORT.md
APP_DATA_REPORT_REQUIREMENTS.md
data/*
```

## 10.3 输出

```text
analysis/APP_ANALYSIS_REPORT.md
```

如果用户明确要求，还可生成：

- PDF
- Word
- Excel
- 演示文稿
- 图表文件

## 10.4 报告建议结构

1. 执行摘要
2. 分析范围与数据说明
3. 核心指标概览
4. 关键发现
5. 页面与用户路径
6. 功能使用与漏斗
7. 广告表现
8. 订阅和 Paywall
9. 投放与回收
10. 版本与策略影响
11. 问题优先级
12. 行动建议
13. 验证方案
14. 数据限制与待补信息
15. 附录：指标口径与数据来源

## 10.5 报告内容规则

最终报告必须：

- 先给结论，再给证据
- 标明分析时间范围
- 标明数据来源
- 标明指标口径
- 标明样本量或用户规模
- 区分事实、推断和建议
- 对建议写明预期影响和验证指标
- 对不确定结论写明置信度
- 对缺失数据保持透明

## 10.6 阶段边界

报告阶段主要负责组织和表达，不应无依据地增加新的计算结论。

如果发现：

- 数字不一致
- 口径冲突
- 引用文件缺失
- 结论无法追溯

应回到 `04-analyze-app.md` 修正分析结果，而不是直接在最终报告中修补。

## 10.7 完成条件

- 报告中的数字与分析结果一致
- 关键结论均有数据依据
- 建议与发现一一对应
- 数据限制已披露
- 最终报告可以独立阅读
- 未引用无法核验的数据

# 11. 数据事实优先级

当不同来源发生冲突时，按以下顺序处理：

```text
原始平台报表或事件明细
  >
已校验的标准化数据
  >
用户明确确认的业务事实
  >
当前项目代码
  >
历史分析报告
  >
文件名或页面名称推断
```

需要注意：

- 代码只能说明逻辑存在，不能说明实际发生次数。
- 用户填写可以解释业务规则，但不能替代平台数据。
- 历史报告只能作为参考，当前分析必须以当前数据为准。
- 来源冲突时必须保留冲突说明，不得静默选择有利于结论的一方。

# 12. 文件更新与覆盖规则

## 12.1 默认规则

- 不覆盖用户上传的原始报表。
- 不覆盖用户已经填写的输入内容。
- 不直接修改 workflow 文件。
- 不把分析结果写回原始数据文件。

## 12.2 重新生成规则

如果目标文件已经存在：

- 扫描报告可在确认后更新，或生成 `_V2` 版本。
- 数据需求文件应保留用户已经确认的映射。
- 用户输入表不得清空已填写内容。
- 数据校验报告可以重新生成。
- 分析结果可以重新生成，但应标明分析时间。
- 最终报告可生成版本号或日期后缀。

建议命名：

```text
APP_ANALYSIS_REPORT_2026-07-14.md
APP_ANALYSIS_REPORT_V2.md
```

# 13. 数据安全规则

整个流程必须遵循：

- 只读取完成任务所需的项目文件。
- 不输出完整密钥、Token、Secret 或证书。
- 不记录个人身份信息，除非分析明确需要且用户已提供。
- 分析示例尽量使用聚合数据，不输出不必要的用户级明细。
- 原始文件保持只读。
- 不执行破坏性命令。
- 不上传本地数据到第三方服务，除非用户明确要求。

# 14. 错误与阻塞处理

遇到错误时，不得假装流程已完成。

## 14.1 缺少前置文件

说明缺少的文件，并指向对应前置 workflow。

例如：

```text
缺少 APP_DATA_VALIDATION_REPORT.md。
当前不能直接执行正式分析，请先运行 workflows/03-validate-input-data.md。
```

## 14.2 数据无法读取

记录：

- 文件名
- 文件格式
- 错误原因
- 是否影响 P0 分析
- 用户需要采取的最小动作

## 14.3 数据不足

允许输出部分结果，但必须说明：

- 已完成哪些模块
- 跳过哪些模块
- 缺少哪些字段或报表
- 当前结论适用范围

## 14.4 口径冲突

同时保留不同口径，不得自行合并。

应说明：

- 冲突指标
- 来源 A 的定义
- 来源 B 的定义
- 对结果的影响
- 需要用户确认的问题

# 15. 专题分析模式

当完整流程已经完成，用户可直接提出专题问题，例如：

- 为什么某页面进入人数下降？
- 哪个页面的流失最大？
- 人均插屏展示次数为什么降低？
- Native 广告位展示率是否异常？
- 某次版本更新是否影响订阅转化？
- 哪些功能使用率最低？
- 某个国家的 ROAS 为什么变化？

专题分析时：

1. 先读取已有扫描、输入、校验和分析文件。
2. 只读取与问题相关的数据文件。
3. 检查当前数据是否覆盖问题所需时间和维度。
4. 必要时先执行局部数据校验。
5. 输出到：

```text
analysis/topics/<TOPIC_NAME>.md
```

6. 不重复生成完整总报告，除非用户要求更新总报告。

# 16. 全流程执行协议

当用户要求运行完整流程时，按以下顺序执行：

```text
Step 1
读取 workflows/01-code-scanner.md
生成 APP_PROJECT_SCAN_REPORT.md

Step 2
读取 workflows/02-design-data-inputs.md
生成 APP_DATA_REPORT_REQUIREMENTS.md
生成 APP_USER_INPUT_REPORT.md

Step 3
等待或检查用户提交的数据与补充信息

Step 4
读取 workflows/03-validate-input-data.md
生成 APP_DATA_VALIDATION_REPORT.md

Step 5
只有在校验允许的范围内，
读取 workflows/04-analyze-app.md
生成 analysis/APP_ANALYSIS_RESULTS.md

Step 6
读取 workflows/05-generate-report.md
生成 analysis/APP_ANALYSIS_REPORT.md
```

如果流程需要等待用户提交数据，应在 Step 2 后停止，不得虚构数据继续执行。
